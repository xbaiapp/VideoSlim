# VideoSlim M2 压缩功能完整化实施计划

> **For Hermes:** Use subagent-driven-development skill to implement this plan task-by-task. 每个写入者使用独立 Git worktree；先规格审查，再质量审查；最终真机边界验收由用户完成。

**Goal:** 在不进入 M3/M4/M5 的前提下，完成 PRD v1.4 的 F3、F6、F7：完整压缩参数、前台服务后台执行、通知、Partial WakeLock、取消、异常恢复清理、ETA 和完整结果页，并交付仅 arm64-v8a 的 M2 APK。

**Architecture:** Flutter/Provider 继续持有跨平台业务状态；Dart 增加纯函数压缩配置与估算。Android 将转码所有权从 Activity 迁移到同进程 `ProcessingService`，由应用级运行时保存活动/终态快照并向重建后的 Flutter EventChannel 重放；Service 独占 Transformer、通知和 WakeLock。异常退出不恢复编码进度，而是在下次合法启动时依据最小任务日志进行所有权受限、幂等的临时文件与未完成 MediaStore 输出清理。

**Tech Stack:** Flutter 3.44.6 / Dart 3.12；Provider；Kotlin/JDK 17；Android minSdk 26 / targetSdk 36；Media3 1.10.1；MediaStore；Android Foreground Service；平台 `MethodChannel`/`EventChannel`；JUnit/Flutter test。

**Hard scope:** 不实现音频提取、裁剪、trim、历史、批量、目标大小、帧率、旋转、MP3、iOS；不引入 FFmpeg/GPL；不申请 INTERNET、媒体读取或广泛存储权限；不创建 production keystore；APK 不提交 Git。

## 设计决定与安全边界

1. 版本升级为 `1.2.0+3`，包名和 debug 签名身份保持不变。
2. M2 运行服务：API 35+ 使用 `mediaProcessing`，API 29–34 使用 `dataSync`，API 26–28 使用普通 `startForeground`；Manifest 同时声明基础及对应类型权限。
3. Android 13+ 在用户从前台启动任务时申请 `POST_NOTIFICATIONS`；拒绝时不静默启动无法看到进度通知的任务。
4. Service 在收到任务后先 `startForeground`，再获取带超时保护的 Partial WakeLock，最后启动预检/Transformer；所有终态、超时、取消与 destroy 路径幂等释放。
5. Activity/Flutter engine 销毁只解绑 Channel，不取消 Service 任务；同进程重新进入 App 通过 `getTaskSnapshot` 恢复进度/结果。
6. 用户 force-stop、断电或整进程死亡不做断点续传；下次合法 App/Service 启动清理并消除僵尸状态。
7. 恢复日志只保存任务 ID、阶段、随机临时文件名、App 创建的输出 URI/legacy 路径、输出文件名和开始时间；不记录不必要的用户路径或媒体内容。
8. 清理仅允许：`cache/transcode/`、经严格目录/文件名/MediaStore 字段验证的 App 自有未完成输出。API 29+ 已 `IS_PENDING=0` 的成功输出永不删除；非本 App URI/路径永不删除。
9. 6 小时/50 GB 是“已验证后正式保证范围”，不是硬格式极限。任一超限先警告，允许自用用户确认后 best-effort 尝试；空间/编码器不满足仍硬阻止。
10. HDR 输入：API 29+ 请求 Media3 OpenGL HDR→SDR；不支持时明确失败；API 26–28 明确提示该设备系统版本无法完成 M2 色调映射。绝不把 HDR 直接解释为 SDR。
11. HEVC 不可用时，业务层降级 H.264 并将目标视频码率乘 1.5，向用户明确提示；H.264 也不可用则阻止。
12. “画质百分比/目标大小/CBR/CQ”不进入 M2；M2 自定义仍为目标平均码率 VBR。
