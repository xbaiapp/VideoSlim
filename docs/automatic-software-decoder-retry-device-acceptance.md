# 一次性自动软件解码重试：真机验收清单

> 日期：2026-07-24
> 候选：VideoSlim `1.9.1+26`
> 代码：`8f4e970c8c724a5019bcc0f56cbbddbb47d2fb33`
> 设备基线：Pixel 10 Pro / Android API 37
> 分发边界：Android Debug certificate，仅私有内部测试

## 安装前

- 保留原 `1.9.0+25 / 11f169c` C2 候选和证据，不覆盖其归档。
- 记录新 APK 文件名和 SHA-256，确认安装界面显示 `1.9.1+26`。
- 使用“更新安装”以保留应用数据；若签名不匹配则停止，不要卸载现有应用后继续。

## A. 正常硬件成功路径

用已知可成功的原视频、同一裁剪区间和同一压缩参数重新执行：

- [ ] 新用户任务仍先选择硬件 decoder。
- [ ] 不出现自动兼容重试文案。
- [ ] 原有 HEVC/H.264 encoder 选择、VBR 行为、音频和元数据保持不变。
- [ ] 只生成一个最终文件，文件可播放、裁剪起止正确。
- [ ] 原文件未修改。
- [ ] 成功后 recovery record 已清理。

## B. 实际硬件 decoder failure 时的自动 fallback

该故障具有非确定性；不要通过永久禁用硬件来伪造。若真实测试再次触发结构化硬件读取失败：

- [ ] 不出现确认弹窗或“使用兼容模式重试”按钮。
- [ ] 不返回首页创建第二个用户任务。
- [ ] 当前任务立即显示“硬件读取失败，已自动改用兼容方式从头重试…”。
- [ ] Flutter 进度和前台通知回到准备阶段/0%，不保留旧 attempt 的约 5% 或更高进度。
- [ ] 日志先记录硬件 attempt/结构化 decoder failure，再记录同任务的软件 attempt。
- [ ] 实际 decoder 为软件实现；encoder、码率目标、裁剪、目录、输出名、音频与元数据策略均未改变。
- [ ] 软件 attempt 只发生一次。
- [ ] 最终最多发布一个文件；不存在首次 attempt 的残缺发布。
- [ ] 成功或失败后旧 temp 和 recovery record 均无泄漏。

## C. 软件 attempt 失败和取消

- [ ] 软件 decoder attempt 若失败，显示普通 terminal failure，不再触发第二轮 fallback。
- [ ] 软件 attempt 运行时取消，任务停止并完成清理；不发布文件。
- [ ] cancelling 文案优先于自动 retry 文案。
- [ ] Activity 切后台、锁屏再回来，仍显示同一个 task ID 的当前软件 attempt，不要求确认或再次提交。

## D. 非资格错误

如遇到 encoder、音频、存储、SAF 发布、权限、普通异常或服务 forced finish：

- [ ] 不启动软件 decoder retry。
- [ ] 保持原稳定错误分类和取消/终止行为。

## 回传证据

请分享**该次测试的完整日志**，不要只截最后几行；同时提供：

- 输入文件名/时长；
- 裁剪起止和压缩参数；
- 是否后台、锁屏或有其他播放器竞争；
- 最终输出文件名、大小、时长、可播放性；
- 若触发 fallback，界面和前台通知各一张截图。

## 验收判定

- **PASS：** A 通过；若自然触发 B，则 B/C 约束全部通过且只发布一个文件。
- **PENDING：** A 通过但非确定性硬件故障未复现；这不等于自动 fallback 的真实设备路径已证明。
- **FAIL：** 出现第二用户任务、循环 retry、旧进度未重置、错误类别越界重试、多文件发布、残缺文件或 recovery 泄漏。
